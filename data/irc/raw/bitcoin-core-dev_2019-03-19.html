04:51 < gmaxwell> mischa010: it's not super simple, because script validity depends on verification flags, which are distinct for mempool and in blocks.
04:51 < gmaxwell> mischa010: so simply 'being in the mempool' isn't enough to guarentee validity. (also because mempool requirements e.g. for locktimes and such aren't exactly the same as the blocks the end up in)
04:52 < gmaxwell> also because we really wouldn't want an otherwise largely harmless bug in mempool handling to turn into a serious consensus bug.
04:53 < mischa010> but what's the use of the cache otherwise?
04:53 < mischa010> if theyre not reused for blocks
04:54 < gmaxwell> they are reused for blocks, but the reason it works via a cache is because it can't just use 'in the mempool'
04:54 < gmaxwell> and, additionally, the cache also reduces some attack vectors.. which was the original purpose of the cache.
04:55 < mischa010> so are all signatures only verified once? up to cache size limits?
04:56 < gmaxwell> yes.
04:56 < mischa010> cool
04:56 < gmaxwell> the cache has two parts, one is for validity of all scriptsigs in the entire transaction.
04:57 < gmaxwell> and that cache is by default good for something like a half million txn.
04:57 < sipa> mischa010: there are 2 distinct caches; one for signatures (which is relatively simple and stores (pubkey,signature,message) tuples), and one for transactions that's much more complicated and stores hashes of transactions together with all context data used for its validation)
