00:54 < phantomcircuit> sipa: is the block/headers synchronziation strategy documented anywhere?
00:55 < phantomcircuit> (mostly im interested in the intended operation)
05:10 < sipa> phantomcircuit: it's complicated...
05:16 < phantomcircuit> sipa, that's uh
05:16 < phantomcircuit> sipa, can you explain it to me?
05:29 < sipa> yes
05:30 < sipa> during initial sync, we send a getheaders to one peer only; outside of it we send it to all
05:30 < sipa> from headers responses we learn what blocks peers have
05:31 < sipa> and then in sendmessages determine which blocks to fetch from whom, subject to various conditions
05:31 < sipa> that's the parallel fetch
05:32 < sipa> there is also the direct fetch, in response to a block 'inv', which (a) sends a getheaders from that inv (b) sends a getdata if we believe we're close to being synced
05:33 < sipa> and then there is bip130, which introduces direct fetch in response to headers
05:34 < sipa> phantomcircuit: for more detail i need to look at the code
05:36 < phantomcircuit> sipa, ok that's helpful
05:37 < sipa> an invariant is that we never send a second request for a block that is already in flight, but we have timeouts that will disconnect (but not ban) a peer that takes too long
05:41 < gmaxwell> with compact blocks that could potentially be relaxed in the future. fetching a redundant compact block is not going to push you over being able to keep up with the network.
05:41 < gmaxwell> nor would a redundant blocktxn so long as it was less than half the block worth of data.
