19:09 < wumpus> #topic Keep reorgs fast for SequenceLocks checks 7187
19:09 < morcos> The idea behind 7187 is that checking sequence locks to determine whether the tx is valid requires looking up the heights of all its inputs. In a reorg, as it stands now, this will require reevaluating the inputs of every single tx in the mempool
19:10 < morcos> That is obviously painful
19:10 < wumpus> yes, that sounds expensive
19:10 < morcos> 7187 attempts to cache for each tx the block hash of the latest block containing an input which had a sequence lock
19:11 < morcos> In the event of a reorg, if that hash is still on the chain, you know the previously calculated height and time (also cached) are still valid.
19:11 < morcos> thus ideally you won't have to look up most txs
19:11 < wumpus> sounds sensible
19:11 < morcos> so i think we should backport that to at least 0.12
19:11 < btcdrak> morcos: without a doubt.
19:11 < wumpus> yes
19:11 < morcos> it's not clear to me whether its worth the effort for 0.11, which already is way slower
19:12 < morcos> it'll still be functionally correct without it
19:12 < morcos> just reorgs will be slow
19:12 < wumpus> probably not - better to keep the patch for 0.11 as small and simple as possible
19:12 < wumpus> it will get less testing after all
19:13 < morcos> so now we need to review/test 7187 and get it merged?
19:13 < wumpus> and with all the changes to the mempool in 0.12 it will probably be non-trivial to backport that optimization
19:13 < morcos> wumpus: thats what i'd be afraid of
