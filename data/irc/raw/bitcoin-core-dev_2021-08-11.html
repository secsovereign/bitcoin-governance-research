08:54 <jnewbery> roasbeef: > since the "new" protocol essentially uses the space between version and verack for interactive protocol feature negotiation instead of service bits
08:54 <jnewbery> My mental model is that service bits are for permanent attributes of the node. They communicate what services the node is capable of providing (historic blocks, segwit serialized blocks & txs, compact block filters, etc), and that service support is gossipped around the network in addrs.
08:54 <jnewbery> That allows nodes that require specific services to find peers that support those services.
08:55 <jnewbery> Feature negotiation using individual messages like sendcmpct, wtxidrelay, sendaddrv2, etc are for features for that specific connection. It's perfectly fine for a node to negotiate those features on some of its connections and not on others.
08:55 <jnewbery> If you need to support addrv2 and not wtxidrelay, it's safe to use p2p protocol version 70016 and just not send `wtxidrelay` during the version handshake. The peer will see that you don't support wtxidrelay and won't inv txs to you using wtxid.
